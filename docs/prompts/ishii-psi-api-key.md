# 石井さんへの依頼: PageSpeed Insights の APIキーを取る（P1.9・急ぎではありません）

**無料**です。作業は Google Cloud Console で5分ほど。

---

## なぜ要るか

デプロイ前後の性能ゲート（`npm run perf:gate`）が **PageSpeed Insights** を使います。
過去の TTFB スパイク事故は**テーマ更新に紛れて入った**ので、更新のたびに測れば当日中に気づけます。

★**キーが無いと測れません。** キー無しの場合、Google の**共有プロジェクト**の
日次クォータを使うため、他人の利用に巻き込まれて 429（上限超過）になります。
実際 2026-07-24 に叩いたら、こう返ってきました。

```
Quota exceeded for quota metric 'Queries' and limit 'Queries per day'
of service 'pagespeedonline.googleapis.com'
for consumer 'project_number:583797351490'   ← 自分のプロジェクトではない
```

キーを取ると**自分のプロジェクトのクォータ**を使うので、他人の影響を受けません。

★このAPIが止まると「更新して重くなったか」が測れなくなるため、
コスト画面（無料枠のツール）にも登録済みです（§4-60）。

---

## 手順

### STEP 1: Google Cloud Console を開く

**認証情報のページ**を直接開きます。

```
https://console.cloud.google.com/apis/credentials
```

★Google アカウントは、GSC / GA4 で使っているものと**同じにしてください**
（別アカウントでも動きますが、後で誰の持ち物か分からなくなります）。

### STEP 2: プロジェクトを選ぶ（または作る）

画面上部にプロジェクト選択があります。

- **既にプロジェクトがある**（GSC のサービスアカウントを作ったもの等）→ それを選ぶ
- **無い / 分からない** → 「新しいプロジェクト」を作る。名前は `mms` などで構いません

### STEP 3: PageSpeed Insights API を有効にする

★**キーを作る前にこれをやってください。** 有効化していないとキーがあっても使えません。

```
https://console.cloud.google.com/apis/library/pagespeedonline.googleapis.com
```

を開いて **「有効にする」**（Enable）を押します。
既に有効なら「管理」と出ているので、そのままで構いません。

### STEP 4: APIキーを作る

1. STEP 1 の**認証情報**ページに戻る
2. 上部の **「＋ 認証情報を作成」** → **「APIキー」**
3. キーが表示されるのでコピー

★このキーは後からでも見られます（Cloudflare のトークンと違い、一度きりではありません）。

### STEP 5（推奨・任意）: キーの用途を絞る

作った直後は「制限なし」です。安全のため絞っておくと安心です。

1. 作ったキーの右の **鉛筆アイコン**（キーを編集）
2. **「APIの制限」** → **「キーを制限」** を選ぶ
3. リストから **「PageSpeed Insights API」** だけにチェック
4. 保存

★**「アプリケーションの制限」（IP制限など）は設定しないでください。**
MMS は自宅/社内のネットワークから叩くので、IPが変わると動かなくなります。

## 私に渡していただく形

`.env` に**石井さんご自身で**追記してください（値を私に貼る必要はありません）。

```
~/システム開発/Next/media-management-system/.env
```

の一番下に1行足します。

```
MMS_PSI_API_KEY=ここにコピーしたキーを貼る
```

> ★`=` の前後にスペースを入れない／`"` で囲まない。

保存したら、これで反映されます。

```bash
cd ~/システム開発/Next/media-management-system && docker compose up -d web
```

★`.env` は git にコミットされません。`.env.example` にはキー名だけ入っています。

## 終わったら教えてください

私の方で実際に測って確認します。

```bash
npm run perf:gate -- before key-test
```

**期待する結果**（4対象すべて数値が出る）

```
デプロイ前の計測: key-test（4件）
  OK  サイトのトップ（テーマ）          TTFB 220ms / LCP 2.10秒
  OK  代表記事（計測タグ）             TTFB 240ms / LCP 2.45秒
  OK  LP: 防災防犯ライト（商材別）      TTFB 90ms / LCP 1.30秒
  OK  LP: 決算前 法人節税対策診断（総合窓口） TTFB 310ms / LCP 2.80秒
```

いまは全部 `NG ... PSI が測れませんでした` になります。

これが通ると **P1.9（性能ゲート）と P3.4（ページ体験の計測）が同時に埋まります**。

## うまくいかないときの見分け方

| 出方 | 意味 | 直し方 |
|---|---|---|
| `API_KEY_SERVICE_BLOCKED` / `has not been used` | **STEP 3 の有効化をしていない** | STEP 3 をやる（有効化の反映に数分かかることがあります） |
| `API key not valid` | キーの貼り間違い | 前後の空白・改行が入っていないか確認 |
| `Requests from referer ... are blocked` | STEP 5 で**アプリケーションの制限**をかけた | 制限を「なし」に戻す |
| `Quota exceeded ... project_number:583797351490` | **キーが読まれていない**（`.env` 未反映） | `docker compose up -d web` を実行したか確認 |

★どれが出たかを教えていただければ、こちらで切り分けます。

MMS が叩くのは**デプロイ前後に4対象ずつ**だけです（1回のデプロイで最大8リクエスト）。
無料枠で十分収まります。日次で自動的に叩くようなことはしていません。
