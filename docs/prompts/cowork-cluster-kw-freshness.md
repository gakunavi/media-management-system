## Q2. メインKWの「正」は yaml と media.db のどちらですか

2箇所に別々に存在します。

| 出所 | 件数 | 備考 |
|---|---|---|
| `shared/keywords/art-kw-map.yaml` | **112件** | 「SEOキーワードの唯一の正」と明記。`main_kw` / `sub_kw` / `search_intent` / `volume_tier` / `difficulty` 付き |
| `tools/media-console/media.db` の `articles.main_kw` | **157/157件** | 全記事に入っている。付随情報なし |

### 聞きたいこと

1. **どちらを正とすべきですか。**
2. **media.db にあって yaml に無い約45件**は、どこで決めた値ですか。
   （yaml が正なら、この45件は yaml へ追記すべき漏れ、という理解で合っていますか）
3. 両者で **main_kw が食い違っている記事**があれば、どちらが新しいですか。
   （こちらで突合できますが、意図的に変えたのか単なる同期漏れなのかが判断できません）

### MMS 側でやろうとしていること（参考）

MMS には GSC の実測データ（記事×検索語）が入っています。そこで **2つを別々に持ちます**。

- **狙い** ＝ 人が決めた `main_kw`（今回移行したいもの）
- **実測** ＝ GSCで実際にクリックを集めている検索語

**この2つのズレをリライト対象の発見に使います。** 例えば ART-177 は「即時償却 税額控除」を狙っていますが、
実測では別の語でしか表示されていません。狙いを持たないとこの比較ができないため、移行をお願いしています。

## Q3. 鮮度（FRESHNESS_TIER）は記事ごとに記録されていますか

`PRJ-030` の設計書にこう書かれていました。

> `config.yaml` に任意フィールド `FRESHNESS_TIER:`（news/commercial/pillar/reference）を追加し、
> ケイデンスを機械判定可能にする

**しかし `FRESHNESS_TIER` を全文検索したところ、この設計書1ファイルにしかヒットしませんでした。**
実装されていない、という理解で合っていますか。

### 聞きたいこと

1. **記事ごとの鮮度階層はどこかに記録されていますか。** 無ければ「無い」で結構です。
2. 記事本文の **「last reviewed: YYYY-MM」** は実際に入っていますか。入っている記事の割合はどのくらいですか。
   （MMS 側で「次回見直し期限」を計算するのに、最終レビュー日が要ります）
3. 無い場合、MMS 側で下記のルールで機械的に付与しようと考えています。**問題ないですか。**

| 鮮度階層 | 更新間隔 | MMS側の判定ルール（案） |
|---|---|---|
| 速報・税制改正 | 随時 | 記事の型 = 時事 |
| 商用・税制系 | 60〜90日（既定75日） | 記事の型 = 商材 / 比較 / リスク |
| エバーグリーン・Pillar | 6ヶ月 | Pillar 記事 or 記事の型 = 実務 |
| 定義・リファレンス | 12ヶ月 | 記事の型 = 制度 |

※「記事の型」は MMS 側でタイトルから機械的に分類済み（商材18 / 比較21 / 制度21 / 時事21 / 実務21 / リスク46 / 事例7 / 未分類24）。

## 補足: 鮮度の根拠は確認済みです（回答不要）

`knowledge/seo/2026-06-09_seo-llmo-research-latest.md §2.6` を読み、

## 回答の形式

箇条書きで構いません。**一覧を出せるものは CSV でいただけると、そのまま取り込めます。**

```
ART番号, クラスタ名, 役割(Pillar/Cluster), メインKW, 鮮度階層
ART-009, P1 即時償却, Cluster, 即時償却 税額控除 違い, commercial
```

繰り返しになりますが、**不明なものは空欄か「不明」**としてください。
MMS 側は空欄を「未入力」として画面に出す設計なので、埋まっていないこと自体が可視化されます。
推測で埋めるより、空欄のほうが安全です。
