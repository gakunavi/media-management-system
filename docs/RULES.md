## 1. ★計測タグの設計原則（7項目・絶対）

> 由来: §3.10.3。**過去にサイト重量化事故（TTFBスパイク）を起こしている。**
> 原因は WPテーマの自前PV計測が**同期でDB書込み**していたこと。
> **本当に問題だったのは「何千回発火しても誰も気づかなかったこと」。同じ設計にしない。**

| # | 規約 |
|---|---|
| **① 1ページあたりの送信は原則1回** | イベントは `sessionStorage` にバッファし、`beforeunload` / `visibilitychange` で **`sendBeacon` 1発**でまとめてPOSTする |
| **② スクロールは throttle 250ms** | 深度は **25 / 50 / 75 / 100% の4段のみ**。**連続値を送らない**（スクロールは1操作で数百回発火しうる） |
| **③ 冪等キー** | `(sessionId, step, contentItemId, 分単位のtimestamp)`。**重複は受信側で捨てる** |
| **④ 送信は必ず非同期** | `sendBeacon` / `fetch keepalive`。**同期XHR禁止** |
| **⑤ WordPress 側では一切DB書込みをしない** | 受けるのは MMS の `/api/ingest`。**`post_meta` / `wp_options` を太らせない** |
| **⑥ 計測JSは defer / 遅延読込** | **`document.write` 禁止** |
| **⑦ 1セッションあたりのイベント上限を JS 側にも置く** | 例: 50件。超えたら送信を止める（**暴走の自己遮断**） |

### 1.1 サーバー側の二重の歯止め

- `/api/ingest` に**レート制限**を置く（同一セッションから毎分N件超は **429**）
- `TelemetryVolume` で**発火回数そのものを監視**する（§3.10.4）

| 閾値（初期値・調整可） | アクション |
|---|---|
| `eventsPerSession > 30` | 段7に**黄**（想定は7〜15） |
| `eventsPerSession > 50` | 段7に**赤** ＋ 即通知 |
| 前日比でイベント総数が **3倍** | **赤 ＋ 計測タグの自動無効化を提案**（段5にワンクリック停止Action） |
| `rejectedDuplicates` 比率 > 20% | 実装バグの疑い。**黄** |

### 1.2 その他の負荷関連の規約

| 対象 | 規約 | 由来 |
|---|---|---|
| A/B の群割当 | **サーバー側で決定する**（JS描画だとちらつき＝CLS悪化） | §3.10.2 |
| Microsoft Clarity | 外部JS。**非同期・遅延読込**にする | §3.10.2 |
| `LinkCheck` | **週次・レート制限つき**（自社サイトへの定期リクエスト） | §3.10.2 |
| `UptimeCheck` | **軽量 HEAD** リクエスト | §3.10.2 |
| WP書き込み（P1.8） | **バッチ化・レート制限**する | §3.10.2 |
| 計測タグの設置場所 | **子テーマ or 専用プラグイン化。`functions.php` 直書き禁止**（テーマ更新で消える） | §16.6-5 |

### 1.3 デプロイ前の性能ゲート（§3.10.5）

- WPテーマ・計測タグ・LPを更新する**前後で必ず PageSpeed Insights を測る**（`PerfGate`）
- **TTFB が 20% 以上悪化、または LCP が 0.5秒以上悪化したらデプロイを失敗扱い**にして段7に赤
- **ロールバック手順を必ず併記**する（`ContentVersion` / テーマzipの旧版）

> **過去の事故はテーマ更新に紛れて入った。更新のたびに測れば、次は当日中に気づく。**

## 2. ★★ 欠測とゼロの区別（絶対・最重要）

> 由来: §3 `MeasurementCoverage` のコメント。
> **この規約を破った結果、「問い合わせ 0」と表示して直客2件の成約を見逃した。**

```
MeasurementCoverage に行が存在しない  = 「未計測」
value = 0                            = 「実測ゼロ」
```

| # | 規約 |
|---|---|
| **2-1** | **UI/API は「未計測」を `—(未計測)` と表示し、決してゼロと書かない** |
| **2-2** | `MeasurementCoverage` に期間が無い指標は、**集計・グラフ・brief から physically 除外する**（0として集計に混ぜない） |
| **2-3** | 計測を開始したら必ず `MeasurementCoverage` に `startedAt` を記録する。**記録を忘れると、その指標は永久に「未計測」扱いになる** |
| **2-4** | MCP の `mms_acquisition_brief` は **`coverage` と `dataQuality` を先頭に置く**。AIは必ずこの2つを先に読んでから結論を出す（§15.2） |

### 2.1 欠測の検知（§3.2.2）

| 規約 | 内容 |
|---|---|
| **日次を永久保持** | `ContentMetric` / `KeywordRanking` に日次で入れ、**削除しない** |
| **★GSC APIは16ヶ月しか遡れない** | それ以前はGoogleから取得不能。**自前DBに貯め続けることだけが唯一の長期履歴**。日次ジョブを止めない運用が資産価値そのもの |
| **欠測の自動補填（backfill）** | 日次ジョブは「最終取得日〜昨日」の**欠けている日を毎回チェックして埋める**。1日失敗しても翌日に自動回復する |
| **欠測アラート** | **3日以上の欠測を段7で赤表示**（過去に10日欠測を誰も検知できなかった） |

## 3. provenance / asOf / confidence の扱い

> 由来: §3 `UnitEconomics.provenance` / §3.4.4 `AdSimulation.assumptionSource` / §3.8.3 / §16.5 / §15.2

### 3.1 provenance（数値の出所）

| 値 | 意味 | 扱い |
|---|---|---|
| `measured` | **自社の実測値** | そのまま使える |
| `declared` | **人の申告値**（例: 電話問い合わせの手動記録・`Lead.sourceType=phone_manual`） | 実測と混ぜて集計しない。UIで区別する |
| `estimated` | **推定値／ベンチマーク**（例: LP CVR の業界一般値・検索ボリューム） | **絶対値を信じず、相対比較と推移で使う** |

| # | 規約 |
|---|---|
| **3-1** | **`AdSimulation` は各前提が実測かベンチマークかを `assumptionSource` に必ず明記する**。LP CVR は自社実測が無いため、**初回は必ず `estimated`** |
| **3-2** | 検索ボリュームは推定値（Keyword Plannerの丸め・各ツールの推定）。**絶対値を信じない**（§3.3.1） |
| **3-3** | 電話問い合わせの手動記録は `provenance=declared` として扱う（§3.8.3） |

### 3.2 asOf（いつ時点のデータか）

| # | 規約 |
|---|---|
| **3-4** | 全ての集計値・グラフに「**いつ時点のデータか**」を併記する |
| **3-5** | 鮮度の管理対象: `KeywordResearch.expiresAt`（ラッコ90日）／`ContentItem.nextReviewDue`（§7.5）／`SnsAccountHealth.tokenExpiresAt`（Threads 60日） |
| **3-6** | **GSCは反映遅延2〜3日ある**。「最新日＝昨日」を前提にしない |

### 3.3 confidence（判定の信頼度）

> 由来: §16.5。記事157本では層別すると対照群が数本しか取れず、統計的に判定できない打ち手がある。

| # | 規約 |
|---|---|
| **3-7** | **対照群の最小サンプル基準: 5記事以上 かつ 合計impressions 500以上**。満たさなければ `verdict = inconclusive` |
| **3-8** | **`inconclusive` は「効果なし（neutral）」とは別物**として UI に明示する。混同を許さない |
| **3-9** | **バッチ判定（最有効）**: 同型の打ち手を複数記事にまとめて適用し、`Intervention.batchId` 単位で判定してサンプル数を稼ぐ |
| **3-10** | 対照群が作れない場合はサイト全体トレンドで補正し、**`confidence = low` を付与**する |
| **3-11** | 判定期間の逐次延長: **14日 → 28日 → 56日**。それでも不足なら `inconclusive` で確定 |

## 6. 判定ロジック（Check）— PDCAが閉じる場所

> 由来: §5.3

```
netEffect = (適用後28日の実測) − (適用前28日の実測) − (対照群の同期間トレンド)

対照群 = 同カテゴリ・同 funnelStage で、その期間に手を入れていない記事群
```

| verdict | 条件 | 自動アクション |
|---|---|---|
| `positive` | netEffect > 期待効果の下限 | `Learning` 生成 → **同型の打ち手を他記事へ横展開提案** |
| `neutral` | 有意差なし | `Learning` 生成 → 立案の重みを下げる |
| `negative` | 悪化 | `Learning` 生成 → **ロールバック提案** ＋ 同型の打ち手を一時停止 |
| `inconclusive` | サンプル不足 | 判定期間を延長（§3-11） |

| # | 規約 |
|---|---|
| **6-1** | **対照群補正（`controlDelta`）を必ず入れる。** 無いと「季節変動で上がっただけ」を「施策が効いた」と誤判定する |
| **6-2** | **季節性の補正も併用する。** 全記事が同じ季節性を持つ場合、対照群補正は効かない（treatment も control も同時に下がる）。`SeasonalityIndex` で調整し、**段1・段4に季節調整値を併記**する（§3.8.1） |
| **6-3** | **ロールバック先を必ず作る。** `Intervention` の前後に `ContentVersion` を保存する。戻す先が無いと negative 判定に対応できない（§16.3） |
| **6-4** | 記事単位では母数不足（クリック142/週 ÷ 157本 = 週1件未満）。**クラスタ単位／記事群単位で判定する**（§3.5.3） |

### 6.1 母数に応じた検証手法の使い分け（§3.7.0）

> **今の段階で小さなCTA文言のA/Bを回すのは時間の無駄。** LPの構成を大きく変えて before/after で見る方が正しい。

| 母数の段階 | 使う手法 |
|---|---|
| **今（LP訪問 〜100/月・実測 約38/月）** | ① **定性**（Clarity録画・リードへのヒアリング）② **大きく変えて before/after** ③ **クラスタ／記事群単位の判定** |
| 中（100〜1,000/月） | ④ **SEOスプリットテスト**（`SplitTest`・主判定手法）＋ ②③ 継続 |
| 先（1,000+/月） | ⑤ **LP/CTA の A/Bテスト**（`Experimentation`） |

| # | 規約 |
|---|---|
| **6-5** | **SEOスプリットテストは層別ランダム化する**（クラスタ × 順位帯 × funnelStage を揃えてから振り分ける）。無作為に分けると初期条件がズレて結論が歪む |
| **6-6** | **LP/CTA A/B は開始時に必要サンプル数を自動計算し、到達見込みが3ヶ月を超えるなら `underpowered` として起動を拒否する。** その場合は「大きく変えて before/after」を `Intervention` 経由で提案する |
| **6-7** | **Clarity は MMS に統合しない**（設計判断）。`Clarity = 探索（仮説を見つける）` / `MMS = 検証（仮説を定量で確かめる）`。無理に取り込むと保守負担だけ増える |

## 7. ★鮮度3階層ケイデンスと cosmetic 更新の禁止

> 由来: §7.5.1（`content-standards.md` が正典）

| freshnessTier | 対象 | 間隔 | 例 |
|---|---|---|---|
| `breaking` | 速報・税制改正 | **随時**（法改正イベント駆動・intervalDays=0） | ニュース記事・改正大綱 |
| `commercial` | 商用・税制系 | **60〜90日（既定75日）** | 即時償却・経営強化税制・商材比較 |
| `evergreen` | エバーグリーン・Pillar | **6ヶ月（180日）** | 節税の基本ガイド |
| `reference` | 定義・リファレンス | **12ヶ月（365日）** | 用語解説 |

### 7.1 ★cosmetic 更新の禁止（絶対）

| # | 規約 |
|---|---|
| **7-1** | **`dataUpdatedAt`（dateModified）は「実質追記」のときだけ更新する。** cosmetic な変更では絶対に触らない |
| **7-2** | `ArticleReview.outcome = no_change` のときは `dataUpdatedAt` を**更新しない**。`updatedDataUpdatedAt = false` を記録する |
| **7-3** | 記事に **`last reviewed: YYYY-MM`** を明示表示する（`lastReviewedAt`） |
| **7-4** | YMYL記事は **`infoBaseDate`（情報基準日）を記事内に表示**する（必須） |

> **cosmetic更新は逆効果。** 中身が変わっていないのに更新日だけ動かすことを禁じる。

### 7.2 自動化（§7.5.2）

```
nextReviewDue = lastReviewedAt + FreshnessRule.intervalDays

日次ジョブ:
  nextReviewDue − 30日 → reviewState = due_soon → 段1に黄色
  nextReviewDue 経過   → reviewState = overdue  → 段1に赤 ＋ 段5に Action 自動起票
```

**期限以外のトリガー（イベント駆動）**

| トリガー | 検知元 | 起票される Action |
|---|---|---|
| 期限到来 | `nextReviewDue` | `periodic_review` |
| **順位下落**（4週で3位以上低下） | `KeywordRanking` | `triggered_by_rank` |
| **表示はあるがクリック0**（CTR異常） | `ContentMetric` | `title_meta_rewrite` |
| **税制改正・法令変更** | News monitor / `RegulatoryEvent` | `triggered_by_law`（**該当する全記事を一括で対象化**） |
| **競合に抜かれた** | `SerpSnapshot` | `triggered_by_competitor` |
| **KW鮮度切れ**（ラッコ90日） | `KeywordResearch.expiresAt` | `kw_refetch` |
| **AIO非引用**（Hot tierで連続hit無し） | AIO計測 | `geo_reinforce` |

### 7.3 レビューの手順（§7.5.3）

1. 記事内の制度・数値・年度を抽出
2. **一次ソース（国税庁・e-Gov・財務省）と自動照合**
3. 差異があれば `findings` に記録し、修正draftを生成
4. 石井さんは段5で**承認するだけ**
5. `outcome` を記録（`no_change` なら `dataUpdatedAt` は更新しない）
6. 実質追記なら `Intervention` に接続して効果測定へ

## 8. WordPress との関係

★ WordPress への書き込みは、必ず MMS API を経由する

### 8.2 ★記事本文の正は WordPress（§16.6-8）

| # | 規約 |
|---|---|
| **8-4** | **本文の正は WordPress。MMS が持つのはメタ・計測・バージョン。** 二重管理しない |
| **8-5** | `ContentVersion.bodyHtml` は**ロールバック用のスナップショット**であり、編集の正ではない |

## 13. 計測の正しさをどう担保するか

> 由来: §16.1。**計測が誤れば、その上の分析・判定・戦略・広告投資が全部無意味になる。**

| # | 対策 | 規約 |
|---|---|---|
| **①** | **合成モニタリング（★これだけは必ず入れる）** | **日次で Playwright が自社サイトを実際に巡回**（記事 → CTA → LP → フォーム表示）し、**期待する `FunnelEvent` がDBに記録されたか検証**する。不一致で段7を赤にする |
| **②** | **二重計測の突合** | GSCクリック vs 自前セッション数、GA4 PV vs 自前PV の**乖離率を日次監視**。**±15%超で警告**。★**どちらが正しいかは判断せず「ズレている」ことを検知する** |
| **③** | **ボット除外** | UA・IP・行動パターン（滞在0秒・step順序が不正）でフィルタ。**除外率そのものも監視**（急増＝実装異常のサイン） |
| **④** | **冪等キーで重複排除** | `(sessionId, step, contentItemId, occurredAt秒)` でユニーク制約。**二重計測を物理的に防ぐ** |
| **⑤** | **外れ値検知** | 前日比・4週平均比で10倍／1/10 等を検出。**欠測検知とは別軸**（「ゼロになった」だけでなく「異常に増えた」も拾う） |
| **⑥** | **集計ロジックの単体テスト** | ゴールデンデータ（既知の入力→既知の出力）で集計関数をテストする |

## 15. 自動化してよいもの／人の承認を残すもの

> 由来: §12.3。**設計思想: 「AIが判断材料を完成させ、人は3秒で押す」。**
> 石井さんの作業時間はほぼゼロになるが、**責任の所在は人に残す。**

| 対象 | 判定 | 残す理由 |
|---|---|---|
| **WP公開の最終切替** | 🔷 **人が押す** | YMYL（税務）。誤情報の公開は税理士法・景表法リスクに直結し取り返しがつかない。**バリデータ全通過＋AI照合表付きで「読んで押すだけ」まで削るが、押すのは人** |
| **一次情報ファクトチェックの承認** | 🔷 **人が押す** | AIが照合表を作ることと、その表が正しいと判断することは別 |
| **DM/リプの個別税務相談・批判への返信** | 🔷 **人が押す** | 税理士法（個別税務助言の禁止）＋レピュテーション |
| **CAPTCHA** | 🔸 **AI突破は禁止** | Bot検知ポリシー違反。**アカウントBANは事業の即死**。検出時のみ通知し、それ以外は自動 |
| **撤退・大型投資の判断** | 🔷 **人が押す** | 経営判断そのもの |
| `BrandMention` の negative | 🔷 **通知のみ** | **AIは自動返信しない**（§3.9.5） |

| # | 規約 |
|---|---|
| **15-1** | 上記以外の手動ステップは**完全自動化する**（§12.2 の14ステップ判定表） |
| **15-2** | **リマインダー方式（「石井さんがローカルで実行してください」）を新規に作らない。** MMS は常駐サーバーなので時間制限・DB書込制限・状態消失が全て解消される |
| **15-3** | `Action.expiresAt` は **14日**。超えたら自動 `expired` にし、段1に「未処理n件」を表示する（§16.6-3） |
