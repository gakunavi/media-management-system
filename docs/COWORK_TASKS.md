# cowork の定期タスク（MMS の外部依存）

2026-07-22 に、cowork 側の Threads 関連の定期スケジュールが削除されていたことが判明した。
これが 7/19 に投稿が止まった直接の原因。ここに再登録用の仕様を残す。

**なぜ MMS のリポジトリに置くか**
cowork は MMS の外にあるが、MMS の入力を作っている。仕様がどこにも書かれていないと、
今回のように「消えたこと自体」が誰にも分からないまま止まる。

---

## 1. 週次: 投稿文の一括生成（月曜）— **要再登録**

### 目的
Threads の投稿キューに1週間分（約110本 = 16本/日 × 7日）の原稿を補充する。

### 出力先
Threads 管理シートの投稿キュー。GAS Web App の `add_bulk` で書き込む。

```
POST {MMS_THREADS_GAS_URL}?key={API_KEY}
{
  "action": "add_bulk",
  "posts": [
    {
      "text": "本文（500字以内）",
      "target": "共通" | "法人" | "個人事業主",
      "core_message": "柱①知識差" | "柱②税理士外" | "柱③早め行動",
      "notes": "フォーマット名 / 生成バッチ名",
      "status": "draft"
    }
  ]
}
```

### ★`status` は必ず `draft`
`pending` で書くと**承認を経ずにそのまま公開される**。
`draft` で入れておけば、MMS の `/threads`「投稿キューの補充」に候補として並び、
石井さんが選んだものだけが `pending` になる（§12.3）。
`status` を省略した場合も `draft` になる（Api.gs の既定値）。

`scheduled_at` は**書かなくてよい**。承認時に MMS が 07〜22時の空き枠を
過去に入らないように割り当てる。

### フォーマット別の実績（2026-07-22 時点・計測済み投稿の平均views）

| フォーマット | 本数 | 平均views |
|---|---:|---:|
| 質問型 | 32 | 1,367 |
| ストーリー | 54 | 1,361 |
| あるある型 | 36 | 1,069 |
| 早口 | 63 | 565 |
| 結論ファースト | 62 | 441 |
| 一言キャッチ | 57 | 437 |
| チェックリスト | 67 | 408 |
| 出題型 | 13 | 390 |
| Good/Bad | 66 | 321 |
| 比較型 | 40 | 311 |

上位3つ（質問型・ストーリー・あるある型）に寄せるのが基本。
ただし**全部を上位3つにはしない** — 下位フォーマットの本数が減ると、
比較の母数が痩せて「効いている/いない」の判定自体ができなくなる。
目安は上位3つで6割、残り4割を他に配る。

★最新の順位は `/threads` の「フォーマット別」で確認すること。
このMDの数字は生成時点のスナップショットで、更新されない。

### テーマの取り方
`/threads` の「次に書くThreads投稿」（型 × テーマ）に、
AIO未引用のキーワードと Threads の反響から起票されたネタが出ている。
ここを見てテーマを決める。

### 本文のルール（YMYL）
以下は GAS 側で機械的に弾かれる。弾かれた行は `error` になり投稿されない。
判定ロジックは `gas/Threads.gs` の `checkYMYL_`（v2.2）が正本。

**禁止（効果の主張と結びついたとき）**
- `必ず節税できる` / `確実に還付されます` / `絶対に節税` / `間違いなく得します`
- `100%節税できる` / `100%安全`
- `保証します` / `guaranteed`
- `あなたは120万円節税できます`（個別シミュレーション）

**禁止（無条件）**
- `知らないと損` / `驚異の` / `ヤバい` / `衝撃の` / `驚愕` / `業界の闇` / `9割が知らない`
- `裏ワザ` / `裏技`（ただし `裏ワザではなく` `裏技より` のような**打ち消し**は可）

**通る（誤検出しない）**
- `取得価額の100%を経費化` / `即時償却100%`（制度上の数値）
- `年内反映を確実にしたいなら`（手続きの話）
- `必ず税理士に確認してください`

**その他**
- 本文は500字以内
- 断定を避け、`〜な場合があります` `〜が現実的です` の語尾を使う

## 2. 日次のチェック — **再登録は不要**

元は「投稿が出ているか・エラーが出ていないか」を毎日見るタスクだった。
**この役割は MMS の `health-alert-daily`（毎朝09:30）が引き継いでいる。**

MMS が毎朝 Slack に投げる内容:

| 検知 | 内容 |
|---|---|
| 配信停止 | 最終投稿からの経過日数 |
| **キュー残数** | 残45本で黄・残15本で赤・0本で「在庫切れ」 |
| GSC欠測 | 日次取得の欠け |
| ツール残高 | DataForSEO 等の枯渇 |
| ジョブ失敗 | worker の失敗 |

**今回の停止も、この仕組みがあれば検知できた。**
週次生成が消えた → キューが減り続ける → 残45本で黄が出る、という順で
「止まる前に」鳴る。日次チェックを人（や cowork）に戻す必要はない。

cowork 側に日次タスクを作るとしたら、MMS から見えないものだけにする。
例: Threads アプリ内にしか出ない警告・アカウント制限の通知。
